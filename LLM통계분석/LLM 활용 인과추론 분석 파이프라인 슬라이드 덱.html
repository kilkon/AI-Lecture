좋은 프롬프트의 4요소 구조

통계 분석 코드 생성을 위해 LLM에 지시할 때는 아래 4가지 요소를 명시적으로 분리하여 제시합니다.

  • 1. Persona (역할 부여)
    "너는 응용미시경제학을 전공한 데이터 과학자야."
  • 2. Context (데이터/맥락)
    "데이터는 패널 구조(unit: state, time: year)이며..."
  • 3. Task (구체적 지시)
    "단일 시점 처리(treatment)를 가정한 DID 회귀 코드를 작성해."
  • 4. Format (출력 형식)
    "pyfixest 라이브러리를 사용하고, 주석을 포함해줘."

💡 Pro-tip

데이터의 전체 행을 입력창에 복사하지 마세요. 대신 df.head().to_markdown()이나 df.info()의 결과만 메타데이터로 제공하는 것이 컨텍스트 창(Context Window) 관리에 효율적입니다.

데이터 정제 전략 (Preprocessing)

통계 분석 데이터의 80%는 정제 과정입니다. 패널 데이터의 특성을 살리는 정제 로직이 필수입니다.

  • 패널 밸런싱(Balanced Panel): 모든 개체(Entity)가 동일한 시점(Time)을 가지도록 결측년도 채우기.
  • 병합(Merge): 처치 시점 데이터(법안 통과일)와 결과 데이터(고용률) 병합.
  • 문자열 처리: 주 이름(State name) 통일화 (예: 'New York' vs 'NY').

경고: 데이터 누수(Data Leakage)

처치 이후(Post-treatment)에 영향을 받는 변수를 통제변수(Covariate)로 정제하여 결측치를 함부로 채우거나 보간(Interpolate)하면 인과적 편의(Bias)가 발생합니다. (이른바 Bad Control 문제)

Agent 기반 통계 자료 수집·검증·시각화 운영 절차

Agent가 웹에서 지표를 찾고 신뢰성 있게 시각화하기 위해, 아래 4단계 운영 규칙으로 작업을 표준화합니다.

1) 자료 찾기 알고리듬

  • 질문을 지표/단위/범위/기간/빈도/비교방식으로 분해
  • KOSIS 우선, 국제비교는 IMF·World Bank 보완
  • 키워드 반복 검색 후 테이블 ID 소범위 샘플 검토
  • 기간·주기·항목의미·코드명확성·결측으로 후보 점수화

2) 자료 검증 알고리듬

  • 스키마(PRD_DE, DT, ITM_ID, C1/C2, PRD_SE, UNIT_NM) 검증
  • 시계열 최소·최대연도, 중복키, 결측률 확인
  • 코드 의미 교차검증(예: 실업률 코드, 고령화지수 혼동 방지)
  • 집계 로직·부분합·전국값 sanity check
  • 차트 값 바인딩, 축/범례/footnote 포함 여부 확인
3) 실무 프롬프트 패턴 (요약)
P1 테이블 찾기: 후보 ID + 기간/주기/단위/항목코드 비교
P2 최종 선정: 기간·주기·범위를 가장 충족하는 1개 선정 근거 3가지
P3 집계 검증: 코드집합 합산 시계열과 원자료 합계 일치 로그 출력
P4 비교 차트: 연도범위 정렬 + 소스/항목코드 footnote 포함
P5 품질 리포트: 기간/결측/중복/단위/코드의미 체크리스트 자동 출력

4) 재현 가능한 운영 규칙

  • 한 파일 누적 원칙: 수집 + 검증 + 렌더 통합
  • 테이블 ID/항목코드/지역코드는 상수로 관리
  • 렌더 직전 period min/max, rows, group count 출력 의무화
  • 모든 차트는 캔버스 내부 footnote로 소스 명시

인과추론 문제정의 (Potential Outcomes Framework)

인과효과 (Causal Effect)

개별 유닛 i에 대해 처치를 받았을 때의 잠재적 결과 Yᵢ(1)과 받지 않았을 때의 결과 Yᵢ(0)의 차이:

τᵢ = Yᵢ(1) - Yᵢ(0)

근본적 문제 (Fundamental Problem)

우리는 동시에 한 개체의 Yᵢ(1)과 Yᵢ(0)을 관측할 수 없습니다. 따라서 적절한 비교군(Counterfactual)을 구성하는 것이 인과추론의 핵심입니다.

연구자의 책임

LLM은 수학적 최적화는 완벽히 수행하지만, "이 대조군이 적절한 Counterfactual인가?"라는 질문에는 답하지 못합니다. 인과적 식별 전략은 연구자의 도메인 지식에 의존합니다.

15

Difference-in-Differences (DID) 직관

시간에 불변하는 개체 고유의 특성(Fixed Effect)과 처리와 무관한 시간의 흐름(Time Trend)을 이중으로 차분하여 순수 효과를 발라냅니다.

그룹 / 시점 Before (Pre) After (Post) Difference 1 (Time)
Treatment (T) Y(T, Pre) Y(T, Post) ΔT = Y(T, Post) - Y(T, Pre)
Control (C) Y(C, Pre) Y(C, Post) ΔC = Y(C, Post) - Y(C, Pre)
Difference-in-Differences (DiD) = ΔT - ΔC
16

DID의 핵심 식별가정 (Identification Assumptions)

1. Parallel Trends (평행 추세)

처치가 없었다면, 처치군의 시간에 따른 결과 변화 추세가 대조군의 변화 추세와 동일했을 것이라는 가정.

*검증 불가능 가정. 단, 처치 이전(Pre-treatment)의 평행 여부를 시각적으로 확인(Event Study)하여 지지함.

2. No Anticipation (사전 반응 없음)

개체가 처치 시점을 미리 알고 처치 이전에 행동을 변경하지 않아야 함.

3. SUTVA

한 개체에 대한 처치가 다른 개체의 결과에 영향을 미치지 않음 (Spillover Effect 부재).

DID Python 구현 1: pyfixest 패키지

Python 생태계에서 R의 fixest와 유사하게 빠르고 고차원의 고정효과(High-dimensional Fixed Effects)를 다루기 위해 최근 각광받는 도구입니다.

import pandas as pd import pyfixest as pf # Two-Way Fixed Effects (TWFE) 모델 추정 # 식별자: unit(주), time(연도) # 종속변수: Y, 처치변수: Treatment_Post (Treat * Post 더미) model = pf.feols( "Y ~ Treatment_Post | unit + time", data=df ) # Cluster Robust Standard Errors 출력 print(model.summary())

DID Event Study 설계 및 해석

처치 전후 시점(t=-2, t=-1, t=0, t=1 ...)별로 효과를 추정하여 사전 평행 추세(Pre-trends)와 동태적 효과(Dynamic effects)를 확인합니다.

Pre-treatment (t < 0)   |   Treatment (t=0)   |   Post-treatment (t > 0)

이상적인 그림: 처치 이전(t < 0)의 계수들은 모두 통계적으로 0(Zero)과 다르지 않아야 평행추세를 지지함. (통상 t=-1을 기준점(Baseline)으로 드랍함)

# pyfixest를 이용한 Event Study 추정 (상대적 시점 변수 time_to_treat 활용) model_es = pf.feols("Y ~ i(time_to_treat, ref=-1) | unit + time", data=df) # iplot() 메서드로 신뢰구간 포함 그래프 바로 렌더링 model_es.iplot(title="Event Study: Dynamic Treatment Effects")

최근 이슈: TWFE의 함정과 Staggered Adoption

개체별로 처치 도입 시점이 다른 경우(Staggered Adoption), 전통적인 OLS 기반 TWFE는 편향된 추정치를 반환할 수 있습니다.

Goodman-Bacon 분해 (2021)

TWFE 추정치는 모든 2x2 DID 추정치의 가중평균입니다. 이때 "이미 처치를 받은 그룹(Already-Treated)"이 "나중에 처치받는 그룹(Late-Treated)"의 대조군으로 쓰이게 됩니다.

처치 효과가 시간에 따라 변한다면(동태적 효과), 이 가중치가 음수(Negative Weight)가 되어 실제 효과와 반대 부호가 나올 수도 있습니다.

해결책 (Modern DID Methods)

  • CS (Callaway & Sant'Anna)
  • Sun & Abraham (pyfixest estimator="saturated")
  • Gardner's 2-Stage (pyfixest pf.did2s())

파이썬에서는 pyfixest가 이러한 최신 방법론들을 빠르게 지원합니다.

DID 결과 해석 시 체크리스트

  • 평행 추세 시각적 확인: 처치 이전 이벤트 스터디 계수들이 0 근처에서 평평한가?
  • 표준오차 클러스터링: 그룹 내 직렬상관(Serial correlation)을 고려하여 Entity 단위로 클러스터링(cluster_entity=True) 하였는가? (Bertrand et al., 2004)
  • 공변량(Covariates)의 외생성: 처치 이후에 처치에 의해 영향을 받을 수 있는 변수(Bad Control)를 모형에 통제하지 않았는가?
  • 도입 시점(Timing): 한 번에 일괄 도입되었는가(Simultaneous), 차례로 도입되었는가(Staggered)? 후자라면 Modern DiD 기법을 사용했는가?

Causal Forest 개요: 평균효과를 넘어 이질성으로

DID가 평균적 정책효과(ATT)에 강하다면, Causal Forest는 누구에게 효과가 큰가라는 질문에 답하기 위한 도구입니다.

CATE 추정

X가 다른 개체마다 처치효과 τ(x)가 어떻게 달라지는지 비모수적으로 추정합니다.

고차원 공변량 대응

상호작용을 사람이 모두 지정하지 않아도 복잡한 비선형 구조를 반영할 수 있습니다.

주의점

이질성 발견이 곧바로 정책 타기팅 근거가 되지는 않습니다. 식별과 해석을 분리해야 합니다.

24

CATE, Orthogonalization, Nuisance Estimation

핵심 개념

  • CATE: 조건부 평균 처치효과 E[Y(1)-Y(0) | X=x]
  • Nuisance Model: outcome model E[Y|X,W], propensity model P(W=1|X)
  • Orthogonalization: 결과와 처치를 잔차화하여 교란의 영향을 약화
  • Cross-fitting / Honesty: 서로 다른 샘플을 써서 편향을 줄이고 일반화 성능을 개선
1. X, T, Y 준비
↓
2. Outcome / Propensity 모델 적합
↓
3. Residual-on-residual 학습
↓
4. 개별 τ(x)와 신뢰구간 해석

Python 구현: EconML CausalForestDML

파이썬에서 가장 널리 쓰이는 구현 중 하나는 EconML의 CausalForestDML입니다.

from econml.dml import CausalForestDML from sklearn.ensemble import RandomForestRegressor, RandomForestClassifier from sklearn.model_selection import train_test_split X = df[feature_cols] T = df['treated'] Y = df['outcome'] cf = CausalForestDML( model_y=RandomForestRegressor(n_estimators=300, min_samples_leaf=10, random_state=42), model_t=RandomForestClassifier(n_estimators=300, min_samples_leaf=10, random_state=42), n_estimators=1000, min_samples_leaf=20, max_depth=None, discrete_treatment=True, random_state=42 ) cf.fit(Y=Y, T=T, X=X) tau_hat = cf.effect(X) lb, ub = cf.effect_interval(X, alpha=0.05)

Causal Forest 결과 해석의 원칙

무엇을 볼 것인가

  • τ̂(x)의 분포: 평균만이 아니라 분산, 꼬리, 음/양 효과 공존 여부
  • 상위 quantile 집단의 특성: 효과가 큰 집단의 공변량 패턴
  • Feature importance: 설명용 단서이지 인과적 메커니즘 증거는 아님
  • Calibration: 예측된 이질성이 holdout 샘플에서도 유지되는가

해석상 주의

Causal Forest는 이질적 효과를 탐색하는 데 강하지만, 특정 변수 하나가 효과 차이의 원인이라고 단정하는 도구는 아닙니다. 상호작용 구조를 요약하는 모델이지, 자동으로 구조적 메커니즘을 밝혀주지는 않습니다.

LLM 환각

존재하지 않는 패키지 함수, 잘못된 인자 이름, outdated API 예시를 제시할 수 있습니다.

코드 실행 = 정답 아님

모형이 돌아가더라도 estimand가 틀리거나 bad control이 들어가면 결론은 잘못됩니다.

과해석

Feature importance나 subgroup pattern을 곧바로 메커니즘으로 읽으면 위험합니다.

핵심 메시지

LLM은 생산성을 올려주지만, 식별가정 검토·결과 검증·학문적 책임은 자동화되지 않습니다.

결론: 박사과정 연구자를 위한 실전 원칙

원칙 1–3

  • LLM에게는 전체 분석이 아니라 작게 쪼갠 작업을 맡긴다.
  • DID는 식별 전략, Causal Forest는 이질성 탐색 전략으로 구분한다.
  • 프롬프트에는 데이터 구조, 가정, 출력 형식을 명시한다.

원칙 4–5

  • 실행 가능한 코드보다 올바른 연구 설계를 우선한다.
  • 모든 결과는 human-in-the-loop 검증과 재현성 문서화를 거친다.

한 문장 요약

LLM은 연구자의 판단을 대체하지 않고, 판단을 더 빠르고 정교하게 실행하도록 돕는 코파일럿입니다.
